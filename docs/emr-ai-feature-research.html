建议先把 AI 做成“文书质控 + 摘要问答 + 草稿助手”，再逐步扩展到 HIS/EMR 全量临床智能。

市面大型 EHR/HIS 的 AI 方向正在收敛：环境语音生成病历、基于 EHR 的图表回顾与问答、病历摘要、医嘱/随访建议、消息与患者沟通、文书质量控制、编码和运营辅助。 结合当前项目现状，第一阶段更适合围绕本地 XML 文书、DCWriter 渲染和打印归档流程做低风险闭环，不宜直接做诊断或医嘱自动执行。

P1 最先落地：文书摘要、质控、问答、打印前检查
P2 接入 HIS/FHIR 后：纵向病程摘要、CDSS、随访和编码
P3 需要治理成熟：环境语音、医嘱建议、风险预测
0 第一阶段不建议自动写回正式病历或自动下医嘱
当前仓库的能力边界是：XML 导入与脱敏、结构化/外部 Canvas 渲染、工具栏、打印和打印预览。AI 应作为“旁路助手”读取脱敏文书并返回建议，必须由医生确认后才进入文书。

市场共识

从 Epic、Microsoft、Oracle、AWS、Google、MEDITECH、athenahealth 及国内电子病历厂商公开资料看，AI 不再是单点聊天机器人，而是嵌入电子病历工作流的“临床工作台能力”。

权威资料

以下资料按“市场产品、标准监管、论文证据”分组。结论重点不是复制厂商功能，而是抽象出可迁移到文书系统的能力单元。

大型 EHR/HIS 厂商 产品趋势
  • Epic AI Charting：内置到 EHR，基于就诊对话生成病历草稿，并可排队生成相关医嘱建议。
  • Microsoft Dragon Copilot：覆盖环境对话采集、临床文档草稿、离散临床数据建议。
  • Oracle Health Clinical AI Agent：语音驱动，嵌入 Oracle Health EHR，面向文档、用药、医嘱和护理协同。
  • MEDITECH Expanse：环境听写、跨结构化和非结构化数据搜索、摘要、问答、生成转诊信。
云厂商与平台能力 基础设施
  • AWS HealthScribe：面向医疗软件厂商的临床文档生成服务，组合语音识别和生成式 AI。
  • Google MedLM 提示示例：展示从医患对话生成 H&P 草稿，但强调输出需人工审阅。
  • Vertex AI Search for Healthcare：定位为检索和总结既有医疗信息，不应脱离持证人员审阅用于直接诊疗。
  • athenahealth Ambient Notes：嵌入移动 EHR，就诊录音后生成摘要并放回患者 encounter。
标准、监管与证据 治理要求
  • 国家卫健委电子病历分级评价通知：以电子病历为核心推进医疗机构信息化建设，是国内落地的基本背景。
  • ONC HTI-1：对认证健康 IT 中 AI/预测算法提出透明度和风险管理要求。
  • WHO AI for Health：强调自主性、安全、透明、问责、公平、可持续等原则。
  • JAMIA 环境 AI scribe 研究：45 名医生、3 个月、9629 次使用，提示可减少文档和 EHR 时间，但个体差异显著。

功能整理

下列功能按当前项目可落地程度分层。点击筛选按钮可查看不同阶段推荐。

AI 文书摘要 优先做

从导入 XML 中抽取主诉、现病史、诊断、医嘱、检查检验、手术/护理记录等，生成医生可快速浏览的结构化摘要。

  • 适合当前本地 XML 流程，不依赖完整 HIS。
  • 输出必须带来源段落和字段定位，避免无根据总结。
  • 可用于右侧 AI 面板、打印前确认页、病历交接。

AI 病历质控 优先做

检查文书完整性、时间逻辑、性别年龄冲突、必填项缺失、术语不规范、前后矛盾、占位符未清理等问题。

  • 这是国内电子病历 AI 最容易产生管理价值的方向。
  • 建议先做规则引擎 + LLM 解释，而不是纯 LLM 判断。
  • 质控结果要分严重程度，并支持一键定位到文书位置。

文内问答与定位 优先做

医生可问“患者是否有青霉素过敏”“出院诊断和入院诊断差异是什么”“哪一段提到感染风险”，系统返回答案和引用。

  • 低风险、高体验提升，适合作为 AI 面板核心入口。
  • 只回答当前文书可证实内容，不做外延诊断。
  • 可和 XML 节点路径、段落索引、页面坐标联动。

段落改写与模板补全 优先做

对医生选中的段落做规范化表述、SOAP/H&P 格式化、摘要化、交接班版、患者版说明等。

  • 只生成草稿，不自动覆盖原文。
  • 需要保留修改前后 diff，医生确认后插入。
  • 可从“开始”工具栏新增 AI 按钮或右侧抽屉进入。

打印/归档前 AI 检查 优先做

在打印、打印预览、归档前自动提示缺签名、缺日期、占位文本、异常空白页、表格未填、敏感信息未脱敏等。

  • 贴合当前项目已有打印链路。
  • 可先做“阻断/提醒/建议”三级提示。
  • 对演示系统也有价值，能体现医疗文书严肃性。

纵向病程摘要 接入后做

基于多次就诊、检查检验、用药、手术和护理记录生成时间线摘要，类似 Oracle Chart Review Agent 和 MEDITECH Expanse Navigator。

  • 需要 HIS/EMR 数据源、患者身份、Encounter 关联。
  • 推荐以 FHIR DocumentReference、Composition、Observation、DiagnosticReport 等资源做抽象层。

临床知识库 RAG 接入后做

把院内制度、路径、模板、诊疗指南、药品说明书接入检索增强生成，用于解释质控规则、推荐模板和辅助查询。

  • 必须显示引用来源、版本、发布日期。
  • 知识库与患者事实分开检索，避免混淆“事实”和“建议”。

编码/CDI 辅助 接入后做

根据病历内容提示 ICD 编码候选、DRG/DIP 影响点、诊断依据不足、主要诊断选择风险。

  • 需结合医保、病案首页、手术操作、费用和编码规则。
  • 先做提示和证据定位，不做自动编码提交。

环境语音生成病历 高治理

录制医患对话，进行说话人分离、转写、临床概念抽取、模板化生成，并由医生审阅后入文书。

  • 市场热度最高，但需要患者告知同意、录音安全、噪声和方言处理。
  • 建议在 P1 文书草稿和质控成熟后再做。

医嘱与诊疗建议 谨慎

基于病历和指南给出检查、检验、用药、随访建议。该方向临床价值高，但属于决策支持，必须满足透明、证据、权限和风险控制。

  • 第一阶段不建议自动执行或自动写回医嘱。
  • 至少要有人审、来源属性、规则版本、审计日志和风险分级。

开发路线

推荐按 3 个阶段推进，先拿到可演示、可验收、低风险的 AI 文书闭环，再逐步进入数据整合和临床决策支持。

1

文书 AI 助手 MVP

目标：在当前 XML 文书工作台内新增 AI 面板。

  • 文书摘要、文内问答、质控清单。
  • 打印前检查和建议导出。
  • 只读分析，不自动改原文。
  • 预计 2-3 周可出可演示版本。
2

AI 草稿与闭环编辑

目标：从“看建议”升级到“医生确认后写入”。

  • 选中文本改写、模板补全、摘要转交接班。
  • 建议 diff、采纳/驳回、审计记录。
  • 接通 DCWriter 编辑 API 后再写回。
  • 预计 3-5 周，取决于编辑 API 暴露程度。
3

HIS/EMR 集成智能

目标：从单份文书扩展到患者级上下文。

  • FHIR/接口层接入患者、就诊、检查、用药、诊断。
  • 纵向病程摘要、知识库 RAG、CDSS 提示。
  • 语音病历和医嘱建议进入试点。
  • 预计 2-3 个月以上，依赖接口和治理。

工程落点

当前项目是 Vue 3 + Vite 前端、.NET Minimal API 后端、DCWriter 外部渲染资源。AI 功能建议以旁路服务接入，先不侵入渲染和打印内核。

建议新增前端模块

  • frontend/src/components/editor/AiAssistantPanel.vue：右侧 AI 面板。
  • frontend/src/composables/useAiAssistant.ts：AI 请求、任务状态、取消、缓存。
  • frontend/src/types/ai.ts：建议、引用、风险、采纳状态类型。
  • ImportToolbar.vue：增加 AI 入口按钮或页签，不改变原有打印能力。

建议新增后端模块

  • /api/ai/document-summary：当前文书摘要。
  • /api/ai/quality-check：病历质控清单。
  • /api/ai/document-chat：基于当前文书问答。
  • /api/ai/rewrite：选区改写和模板补全。
  • AiAuditStore：记录输入摘要、输出、采纳动作、模型版本。

安全合规

医疗 AI 的核心不是“能不能生成”，而是“能不能解释、追溯、回滚、审计，并让医生保留最终责任”。建议从第一天就把这些机制做进产品。

人工确认

所有 AI 输出默认是草稿或建议。写回病历、打印归档、用于诊疗决策前必须经过医生确认。

  • 每条建议有“采纳/忽略/修改后采纳”。
  • 保留原文、AI 草稿和最终文本 diff。

来源可追溯

摘要、质控、问答必须带证据位置。不能在文书中找到依据的内容，应标记为“未证实”。

  • 引用 XML 节点路径、段落、页码或渲染坐标。
  • 知识库引用需带版本和发布日期。

隐私与数据最小化

默认只上传完成脱敏后的必要片段。真实医院部署时需要院内部署、专线或合规云，并记录访问边界。

  • 患者标识、手机号、身份证、地址等先脱敏。
  • AI 请求和响应进入审计日志。
风险点 典型表现 产品控制 技术控制
幻觉与过度推断 AI 编出文书不存在的病史、诊断或治疗建议。 “仅基于当前文书回答”；无证据则提示未找到。 RAG 引用校验、答案逐条证据绑定、置信度和拒答策略。
自动写回风险 未经医生确认就改变正式病历。 所有生成内容进入待确认区，医生点击后才写入。 写回 API 权限分离、diff 审计、撤销和版本记录。
临床责任不清 医生误以为 AI 输出等同诊疗结论。 高风险功能标记“辅助建议”，不替代医生判断。 模型卡、来源属性、风险分级、角色权限。
数据泄露 真实病历进入外部模型或日志。 部署说明明确数据边界，默认禁用真实患者信息外发。 脱敏、最小字段、加密传输、审计、可配置模型网关。
